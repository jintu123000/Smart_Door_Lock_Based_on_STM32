一、简历项目描述（复制可用）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

项目名称：基于STM32的多功能智能门锁系统
开发环境：Keil MDK 5 / STM32F103C8T6 / STM32标准外设库
项目时间：2025.xx — 2026.xx

项目描述：
设计并实现了一套嵌入式智能门锁系统，支持RFID刷卡、矩阵键盘密码输入两种开锁方式，
通过蓝牙模块向手机APP发送开锁状态，同时显示环境温湿度等运行信息。

主要工作：
? 基于STM32F103C8设计系统架构，采用TIM2定时器中断（100Hz）实现多任务分时调度，
  统一协调键盘扫描（150ms）、DHT11采集（5s）、RFID检测（800ms）等并发任务
? 使用软件模拟SPI协议驱动RC522 RFID读卡模块，实现完整的ISO14443A/Mifare协议栈，
  包括寻卡、防冲突、UID读取、密码验证、数据块读写等操作
? 驱动128×64 OLED（SSD1306/I2C），实现双缓冲刷新机制以消除闪烁，
  设计双界面切换UI（密码输入界面 / 运行状态界面）
? 通过TIM3 PWM（50Hz）控制舵机执行门锁机械动作，角度控制精度±1°
? 使用USART3（9600bps）对接蓝牙模块，密码验证成功后发送自定义指令"<G>开锁成功"
? 基于DHT11单总线协议实现温湿度采集，实时显示于OLED状态界面

"我做了一个基于STM32F103的智能门锁项目。

核心功能有两个开锁方式：一是RFID刷卡，用的是RC522读卡器，我用软件模拟SPI实现了完整的
Mifare协议，包括寻卡、防冲突、密码认证、读写卡数据块这整套流程；二是矩阵键盘输入6位
密码，密码匹配后通过USART串口给蓝牙模块发指令，通知手机端开门。

整个系统的任务调度我用TIM2定时器中断来实现——10ms中断一次，通过计数器把键盘扫描、温
湿度采集、RFID检测、OLED刷新这些任务分开调度，避免主循环阻塞。舵机用TIM3的PWM控制，
门锁显示模块用的128×64的OLED，我做了双缓冲避免屏幕闪烁。

这个项目让我对STM32外设驱动、中断调度、通信协议有了比较深的理解，也踩了一些RFID时序
和串口冲突的坑。"

【基础层——大概率必问】

Q1：你项目里为什么用软件SPI而不是硬件SPI驱动RC522？
------------------------------------------------------
A：主要有两个原因：
一是RC522对时序要求不算苛刻（最大10MHz），软件模拟完全够用；
二是当时PA4/PA5/PA6/PA7这些硬件SPI引脚被其他外设占用（PA6给了TIM3 PWM），所以改用
PB12~PB15做软件SPI，引脚灵活、不抢资源。
软件SPI的缺点是占CPU，但因为RFID只在800ms定时里触发一次，对性能影响可以忽略。


Q2：TIM2定时器怎么实现多任务调度的？
------------------------------------------------------
A：TIM2配置为PSC=7199、ARR=99，72MHz时钟下定时周期正好10ms，即100Hz频率产生中断。
中断里维护3个计数器：
  - key_num_read_cnt 累计到15（150ms）→ 触发键盘扫描
  - dht11_read_cnt   累计到500（5000ms=5s）→ 设置标志位，主循环读DHT11
  - OLED_read_cnt    累计到80（800ms）→ 设置标志位，主循环刷OLED并触发RFID检测

中断里"只设标志位，不做耗时操作"，实际读取放在主循环里处理。
这样ISR执行时间极短，避免阻塞其他中断。


Q3：RFID的流程能说一下吗？
------------------------------------------------------
A：用的Mifare S50卡，完整流程是：
  1. PcdRequest()    发出寻卡请求，获取卡片类型
  2. PcdAnticoll()   防冲突，从多张卡中选一张，获取4字节UID
  3. PcdSelect()     用UID选定这张卡，进入激活状态
  4. PcdAuthState()  用6字节密钥（默认0xFF×6）对目标块做密码验证（Block 6）
  5. PcdWrite(4,...) 往Block 4写入数据，验证写操作是否正常
  6. PcdRead(4,...)  读取Block 4数据，验证读操作是否正常
  7. 写成功后把密码数组input_key设为{1,2,3,4,5,6}，触发UI层密码匹配逻辑开门

底层通信是软件模拟SPI，高位先行，每个字节8个时钟沿。


Q4：DHT11是怎么读数据的？为什么不用I2C或SPI？
------------------------------------------------------
A：DHT11用的是自定义的单总线协议（1-Wire变种），传感器厂商规定的，不支持标准I2C/SPI。
读取流程：
  1. 主机把引脚拉低≥18ms，发起通信请求
  2. 释放后等DHT11应答（先80μs低电平再80μs高电平）
  3. 循环读40位数据，判断每位是0还是1：
     高电平持续约26-28μs 是0，70μs 是1，在40μs左右采样
  4. 40位 = 湿度高8 + 湿度低8 + 温度高8 + 温度低8 + 校验和
  5. 做完校验验证数据有效性

代码里用的是PC13引脚，通过动态切换GPIO方向实现收发复用。


Q5：舵机控制的PWM是怎么算的？
------------------------------------------------------
A：舵机需要50Hz的PWM，脉宽0.5ms~2.5ms对应0°~180°。

TIM3配置：
  PSC = 71（72分频），计数频率1MHz
  ARR = 19999，周期 = 20ms = 50Hz

角度换算公式：CCR = Angle/180 * 2000 + 500
  0°  → CCR=500  → 脉宽0.5ms
  90° → CCR=1500 → 脉宽1.5ms
  180°→ CCR=2500 → 脉宽2.5ms

引脚PA6是TIM3_CH2的复用引脚，配置为复用推挽输出。


【进阶层——准备好加分】

Q6：密码是写死的吗？有没有持久化存储？
------------------------------------------------------
A：主工程里密码写死在代码中 door_key[6]={1,2,3,4,5,6}，掉电不保存，这是后续可完善的点。

但子工程（key_lock/，基于STM32F0xx）里已经实现了Flash持久化存储——把密码写入Flash的
Page15（地址0x08003C00），掉电重启后从Flash读取，支持通过键盘菜单修改密码后写入Flash。

主工程后续可同样方案实现，先擦除页再写入，STM32内部Flash擦写周期至少1万次。


Q7：蓝牙部分是怎么实现的？
------------------------------------------------------
A：硬件侧是USART3（PB10/PB11）连接蓝牙串口模块（HC-05或类似），波特率9600bps。
开锁成功后发送字符串 "<G>开锁成功" 作为自定义协议帧，<G> 是指令头，方便APP端解析。
APP端用配套蓝牙串口助手接收并展示。
如果要做完整APP，可以用BLE+Android/iOS端开发，换ESP32或nRF52做BLE GATT服务。


Q8：OLED双缓冲是什么意思？为什么要这样做？
------------------------------------------------------
A：双缓冲是指代码操作内存里的"显存数组"（1024字节，128×64/8），而不是直接写屏幕。
所有的ShowString、ShowNum等操作只改这个数组，最后统一调用OLED_Update()通过I2C把整个
数组推到屏幕。

好处是避免闪烁——逐字符发I2C屏幕会明显抖动；整帧刷新视觉更平滑。
代价是每次刷新发1KB数据，所以定时器控制刷新频率约800ms一次，不用太频繁。


Q9：你遇到过什么比较难解决的问题？
------------------------------------------------------
A：有三个印象比较深：

① RFID读卡不稳定：
   初期刷卡偶尔成功偶尔失败，排查发现是软件SPI时序问题——SCK高低切换之间建立时间不足，
   导致MISO数据采样偏移。在时钟切换后加了几个微秒延时后稳定了。

② DHT11被中断打断：
   DHT11数据读取全靠精确延时，如果中间被TIM2中断打断，采样时间点就偏了，读出数据会
   校验失败。解决方案是读取40位数据期间临时关闭TIM2中断，读完再开启。

③ printf串口切换问题：
   系统有两个串口，调试走USART1，蓝牙走USART3，用set_printf_usart()动态切换。
   一开始发送蓝牙指令后忘记切回USART1，导致调试信息全跑到蓝牙模块那边，排查了一段时间。


Q10：这个项目如果继续改进，你会做什么？
------------------------------------------------------
A：有以下几个方向：
  1. 密码存Flash：主工程加Flash持久化，参考子工程方案，用户可修改密码
  2. 错误尝试锁定：输错3次锁定30秒，防暴力破解
  3. RFID白名单：把授权UID存Flash里，只有白名单卡才能开锁，而不是任意卡
  4. BLE远程开锁：换ESP32或nRF52，手机端BLE远程控制门锁
  5. RTOS化：引入FreeRTOS，每个外设独立任务，架构更清晰、更易扩展

【TIM2 调度定时器】
  PSC = 7199, ARR = 99
  中断频率 = 72MHz / 7200 / 100 = 100Hz（10ms/次）
  键盘扫描：每150ms（计数>15）
  DHT11：  每5000ms（计数>500）
  OLED刷新：每800ms（计数>80）

【TIM3 舵机PWM】
  PSC = 71, ARR = 19999
  PWM频率 = 72MHz / 72 / 20000 = 50Hz
  公式：CCR = Angle/180 * 2000 + 500

【RFID默认密钥】
  card_KEY[6] = {0xFF, 0xFF, 0xFF, 0xFF, 0xFF, 0xFF}

【默认开门密码】
  door_key[6] = {1, 2, 3, 4, 5, 6}  （即123456）

【蓝牙开锁指令】
  "<G>开锁成功"  通过USART3发送

【Flash密码存储地址（子工程）】
  Page15 地址：0x08003C00
